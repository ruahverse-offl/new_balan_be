# CRUD Operations Implementation Plan

## Overview
This document outlines the implementation approach for CRUD (Create, Read, Update, Delete) operations for all tables defined in `Yaml_DB.yaml`.

## Database Schema Notes

### Audit Fields (All Tables)
All tables include the following audit fields:
- `id` (UUID) - Primary key, auto-generated by database
- `created_by` (UUID) - User who created the record
- `created_at` (TIMESTAMPTZ) - Creation timestamp
- `created_ip` (VARCHAR(45)) - IP address of creator (supports IPv6)
- `updated_by` (UUID) - User who last updated the record
- `updated_at` (TIMESTAMPTZ) - Last update timestamp
- `updated_ip` (VARCHAR(45)) - IP address of updater (supports IPv6)
- `is_deleted` (BOOLEAN) - Soft delete flag

### Master Tables
Master tables additionally include:
- `is_active` (BOOLEAN) - Active/inactive status

### Transaction Tables
Transaction tables do NOT include `is_active` field.

## CRUD Operations Specification

### 1. CREATE Operation

#### Endpoint Pattern
```
POST /api/v1/{resource_name}
```

#### Request Body
- **Exclude**: `id`, `created_at`, `updated_by`, `updated_at`, `updated_ip`
- **Include**: All business fields + `created_by`, `created_ip`
- **Auto-set**: `is_deleted = false`
- **Auto-set**: `is_active = true` (for master tables only)

#### Implementation Details
- UUID for `id` is generated by database (PostgreSQL UUID default)
- `created_by` extracted from JWT token (current user)
- `created_ip` extracted from request headers (`X-Forwarded-For` or `X-Real-IP` or direct client IP)
- `created_at` set by database (TIMESTAMPTZ default)
- `is_deleted` defaults to `false`
- For master tables: `is_active` defaults to `true`

#### Response
- **Status**: `201 Created`
- **Body**: Created entity with all fields including generated `id` and timestamps

---

### 2. READ Operations

#### 2.1 Get by ID
**Endpoint Pattern**: `GET /api/v1/{resource_name}/{id}`

**Query Parameters**: None

**Response**:
- **Status**: `200 OK` if found, `404 Not Found` if not found
- **Body**: Single entity object
- **Filter**: Automatically exclude soft-deleted records (`is_deleted = false`)

#### 2.2 Get List (with Pagination, Search, Sort)
**Endpoint Pattern**: `GET /api/v1/{resource_name}`

**Query Parameters**:
- `limit` (int, optional): Number of records per page (default: 20, max: 100)
- `offset` (int, optional): Number of records to skip (default: 0)
- `search` (string, optional): Search term to match against ALL searchable fields
- `sort_by` (string, optional): Field name to sort by (default: `created_at`)
- `sort_order` (string, optional): `asc` or `desc` (default: `desc`)

**Search Behavior**:
- Search applies to ALL text/varchar fields in the table
- Case-insensitive partial matching (ILIKE for PostgreSQL)
- Numeric fields: exact match only
- Date fields: not searchable via text search
- Boolean fields: not searchable via text search

**Sort Behavior**:
- Can sort by ANY field in the table
- Default sort: `created_at DESC` (newest first)
- Supports multiple sort fields (comma-separated): `sort_by=name,created_at&sort_order=asc,desc`

**Response**:
- **Status**: `200 OK`
- **Body**:
```json
{
  "data": [...],
  "pagination": {
    "total": 150,
    "limit": 20,
    "offset": 0,
    "has_next": true,
    "has_previous": false
  }
}
```
- **Filter**: Automatically exclude soft-deleted records (`is_deleted = false`)

---

### 3. UPDATE Operation

#### Endpoint Pattern
```
PUT /api/v1/{resource_name}/{id}
PATCH /api/v1/{resource_name}/{id}
```

**Note**: Use `PUT` for full updates, `PATCH` for partial updates (recommended: `PATCH`)

#### Request Body
- **Exclude**: `id`, `created_by`, `created_at`, `created_ip`
- **Include**: Fields to update + `updated_by`, `updated_ip`
- **Auto-set**: `updated_at` by database
- **Optional**: Can update any business field

#### Implementation Details
- `updated_by` extracted from JWT token (current user)
- `updated_ip` extracted from request headers
- `updated_at` set by database (TIMESTAMPTZ default)
- Cannot update `id`, `created_by`, `created_at`, `created_ip`
- Returns `404` if record not found or soft-deleted

#### Response
- **Status**: `200 OK` if updated, `404 Not Found` if not found
- **Body**: Updated entity with all fields

---

### 4. DELETE Operation (Soft Delete)

#### Endpoint Pattern
```
DELETE /api/v1/{resource_name}/{id}
```

#### Request Body
None (ID in URL path)

#### Implementation Details
- Sets `is_deleted = true`
- Sets `updated_by` from JWT token
- Sets `updated_ip` from request headers
- Sets `updated_at` by database
- Does NOT physically delete the record
- Returns `404` if record not found or already soft-deleted

#### Response
- **Status**: `200 OK` if deleted, `404 Not Found` if not found
- **Body**: 
```json
{
  "message": "Resource deleted successfully",
  "id": "uuid-here"
}
```

## Error Handling

### Standard Error Responses

**400 Bad Request**:
- Invalid request body
- Validation errors (Pydantic)
- Invalid query parameters

**401 Unauthorized**:
- Missing or invalid JWT token
- Expired token

**404 Not Found**:
- Resource not found by ID
- Resource is soft-deleted
- Invalid resource ID format

**409 Conflict**:
- Unique constraint violations
- Foreign key violations (if applicable)

**422 Unprocessable Entity**:
- Business rule violations
- Invalid data relationships

**500 Internal Server Error**:
- Database errors
- Unexpected exceptions

### Error Response Format
```json
{
  "error": {
    "code": "RESOURCE_NOT_FOUND",
    "message": "Resource with ID {id} not found",
    "details": {}
  },
  "request_id": "uuid-here",
  "timestamp": "2026-02-01T10:30:00Z"
}
```

## Database Models (SQLAlchemy)

### Base Model Pattern
```python
class BaseModel(Base):
    __abstract__ = True
    
    id = Column(UUID(as_uuid=True), primary_key=True, server_default=text("gen_random_uuid()"))
    created_by = Column(UUID(as_uuid=True), nullable=False)
    created_at = Column(TIMESTAMPTZ, nullable=False, server_default=text("NOW()"))
    created_ip = Column(String(45), nullable=False)
    updated_by = Column(UUID(as_uuid=True), nullable=True)
    updated_at = Column(TIMESTAMPTZ, nullable=True, onupdate=text("NOW()"))
    updated_ip = Column(String(45), nullable=True)
    is_deleted = Column(Boolean, nullable=False, default=False, server_default="false")
```

### Master Table Model Pattern
```python
class MasterModel(BaseModel):
    __abstract__ = True
    
    is_active = Column(Boolean, nullable=False, default=True, server_default="true")
```

## PARKED ITEMS (Questions/Clarifications)

1. **Foreign Key Validation**: Should we validate foreign keys exist before create/update?
   - Example: `role_id` in `users` table - validate `roles.id` exists?

2. **Unique Constraints**: Are there unique constraints beyond what's in YAML?
   - Example: `email` in `users` should be unique?

3. **Cascade Behavior**: What happens on delete?
   - Example: If `role` is deleted, what happens to `users` with that `role_id`?

4. **Search Fields**: Should we exclude certain fields from search?
   - Example: Should `password_hash` be excluded from search?

5. **Sort Fields**: Should we restrict sorting to certain fields?
   - Example: Should we allow sorting by `password_hash`?

6. **Master vs Transaction**: Different behavior for master vs transaction tables?
   - Example: Can we soft-delete master records that are referenced?

7. **RBAC**: Which roles can perform which operations?
   - Example: Can all users create/update/delete, or role-based?

8. **Bulk Operations**: Do we need bulk create/update/delete endpoints?

9. **Audit Trail**: Do we need a separate audit log table for tracking changes?

10. **Soft Delete Recovery**: Do we need an endpoint to restore soft-deleted records?

## Notes

- All UUIDs are generated by PostgreSQL (use `gen_random_uuid()`)
- All timestamps are set by PostgreSQL (use `NOW()`)
- IP addresses support both IPv4 and IPv6 (max 45 characters)
- Soft delete is the default - no hard deletes
- All list queries automatically filter `is_deleted = false`
- Search is case-insensitive and supports partial matching
- Sort supports multiple fields (comma-separated)
- Pagination uses limit/offset pattern (not cursor-based)
